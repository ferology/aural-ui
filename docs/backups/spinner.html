When to Use Spinners

Spinners provide visual feedback during loading states and asynchronous operations:

  • Data fetching: When loading content from an API or database
  • Form submission: During form processing or validation
  • Page transitions: When navigating between views or routes
  • File operations: During file uploads, downloads, or processing
  • Background tasks: For operations that take more than 1 second

Best Practices

Visual Design

  • Use appropriate sizes relative to the container or context
  • Choose spinner variants that match the visual weight of your interface
  • Maintain consistent spinner styles throughout your application
  • Consider using different variants for different contexts (e.g., dots for inline, default for cards)

Color Usage

  • Use --primary for general loading states
  • Use --white for spinners on dark backgrounds (like primary buttons)
  • Use --success for successful operations in progress
  • Use --warning for operations that require attention
  • Use --error sparingly, typically only when retrying failed operations

Timing

  • Show spinners immediately for operations expected to take more than 1 second
  • For very fast operations (under 300ms), consider not showing a spinner to avoid flashing
  • Always include descriptive text for operations longer than 3 seconds
  • For operations longer than 10 seconds, consider progress bars instead

User Experience

  • Always disable interactive elements during loading to prevent duplicate actions
  • Provide context with text labels when possible ("Loading...", "Saving...", "Processing...")
  • For full-page loading, use an overlay to prevent interaction with underlying content
  • Consider skeleton screens for complex layouts instead of just spinners

Accessibility

Ensure spinners are accessible to all users:

  • Add role="status" to spinner containers for screen reader announcements
  • Include aria-label with descriptive text (e.g., "Loading content")
  • For spinners with visible text, ensure text has sufficient color contrast (4.5:1 minimum)
  • Disable interactive elements during loading to prevent confusion
  • For full-page overlays, ensure keyboard focus is managed appropriately
  • Consider aria-live="polite" for dynamic status updates

Example with Accessibility Attributes

<div class="aural-spinner aural-spinner--primary"
     role="status"
     aria-label="Loading content">
    <div class="aural-spinner__circle"></div>
</div>

Mobile Considerations

  • Use larger spinners (at least --sm) on mobile for better visibility
  • Ensure spinner overlays cover the entire viewport to prevent accidental taps
  • Consider performance: use simpler spinner variants (default or dots) for low-end devices
  • Test spinner visibility on various screen sizes and resolutions
  • Avoid very small spinners (xs) as primary indicators on mobile devices
