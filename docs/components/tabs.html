Tabs

Organize content into separate views where only one view is visible at a time. Tabs let users switch between related content groups without leaving the page.

Basic Tabs

Default tab style with underline indicator. The active tab is marked with .tab-active class.

<div class="tabs" role="tablist">
  <button class="tab tab-active" role="tab">Profile</button>
  <button class="tab" role="tab">Settings</button>
  <button class="tab" role="tab">Activity</button>
</div>
Pills Style

Pills style with rounded background for the active tab. Use .tabs-pills for a softer, more modern appearance.

<div class="tabs tabs-pills" role="tablist">
  <button class="tab tab-active" role="tab">Home</button>
  <button class="tab" role="tab">Products</button>
</div>
Boxed Style

Boxed style with contained background. Use .tabs-boxed for a more compact, segmented control appearance.

<div class="tabs tabs-boxed" role="tablist">
  <button class="tab tab-active" role="tab">Dashboard</button>
  <button class="tab" role="tab">Analytics</button>
</div>
Tabs with Icons

Add icons before tab labels to improve scanability and visual hierarchy. Icons help users quickly identify tab content.

<button class="tab tab-active" role="tab">
  <i data-lucide="user"></i>
  Profile
</button>
Tabs with Badges

Use badges to show counts or status indicators within tabs. Particularly useful for messaging, notifications, or filtering interfaces.

<button class="tab tab-active" role="tab">
  <span>All</span>
  <span class="badge badge-sm">24</span>
</button>
Disabled Tabs

Disable tabs that are not yet accessible using the disabled attribute. Common in multi-step forms or workflows where steps must be completed sequentially.

<button class="tab" role="tab" disabled>
  Step 3: Payment
</button>

Complete example showing tabs with corresponding content panels. JavaScript handles switching between panels when tabs are clicked.

<div class="tabs" role="tablist">
  <button class="tab tab-active" onclick="switchTab(event, 'panel1')">
    Tab 1
  </button>
  <button class="tab" onclick="switchTab(event, 'panel2')">
    Tab 2
  </button>
</div>

<div id="panel1" class="tab-panel">Content 1</div>
<div id="panel2" class="tab-panel" style="display: none;">Content 2</div>

<script>
function switchTab(event, panelId) {
  // Remove active class from all tabs
  document.querySelectorAll('.tab').forEach(tab => {
    tab.classList.remove('tab-active');
  });

  // Add active class to clicked tab
  event.target.classList.add('tab-active');

  // Hide all panels
  document.querySelectorAll('.tab-panel').forEach(panel => {
    panel.style.display = 'none';
  });

  // Show selected panel
  document.getElementById(panelId).style.display = 'block';
}
</script>

Best Practices

When to Use Tabs

  • Organize Related Content: Group related information that users don't need to see simultaneously (product details, specifications, reviews)
  • Settings Pages: Separate settings into categories (profile, privacy, notifications, security)
  • Data Views: Provide different perspectives on the same data (table view, chart view, calendar view)
  • Multi-Step Forms: Show progress through sequential steps (though consider a stepper component for required sequential completion)
  • Filtering Content: Switch between filtered views (all items, unread, starred)

When NOT to Use Tabs

  • Content Comparison: Don't use tabs when users need to compare information side-by-side - use panels or columns instead
  • Sequential Tasks: For required step-by-step workflows, use a stepper component that shows progress and prevents skipping
  • Too Many Options: Avoid more than 5-7 tabs - consider a dropdown or sidebar navigation for larger sets
  • Mobile-Heavy Content: Tabs can be difficult on small screens - consider accordions or stacked sections for mobile
  • Critical Information: Don't hide essential information in tabs - users may not explore all tabs

Tab Organization

  • Logical Order: Arrange tabs from most to least important, or in a logical workflow sequence
  • Clear Labels: Use concise, descriptive labels (1-2 words). Avoid vague terms like "Other" or "More"
  • Consistent Width: Keep tab widths balanced - avoid one very long tab next to several short ones
  • Default Selection: Always have one tab active by default (usually the first or most important)
  • Remember Selection: Consider persisting the active tab when users navigate away and return

Accessibility

  • Use role="tablist" on the tabs container
  • Use role="tab" on each tab button
  • Use role="tabpanel" on content panels
  • Add aria-selected="true" to the active tab
  • Connect tabs to panels using aria-controls and id
  • Support keyboard navigation: Arrow keys to move between tabs, Enter/Space to activate
  • Ensure sufficient color contrast for active/inactive states
  • Don't rely solely on color - use underlines, backgrounds, or icons for active state

Mobile Considerations

  • Scrollable Tabs: Allow horizontal scrolling for tabs on small screens rather than wrapping
  • Touch Targets: Ensure tabs are at least 44px tall for easy tapping
  • Alternative Patterns: Consider replacing tabs with accordions or a dropdown select on mobile
  • Visible Indicator: Make it obvious that more tabs are available off-screen (shadow or fade)
