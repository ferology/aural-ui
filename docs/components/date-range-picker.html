Framework Examples

<div class="date-range-inputs">
  <input type="date" class="date-start" placeholder="Start date">
  <input type="date" class="date-end" placeholder="End date">
</div>

<script>
const startInput = document.querySelector('.date-start');
const endInput = document.querySelector('.date-end');

startInput.addEventListener('change', (e) => {
  endInput.min = e.target.value;
});

endInput.addEventListener('change', (e) => {
  startInput.max = e.target.value;
});
</script>
const DateRangePicker = ({ startDate, endDate, onChange }) => {
  const [range, setRange] = useState({
    start: startDate || '',
    end: endDate || ''
  });

  const handleStartChange = (e) => {
    const newRange = { ...range, start: e.target.value };
    setRange(newRange);
    onChange?.(newRange);
  };

  const handleEndChange = (e) => {
    const newRange = { ...range, end: e.target.value };
    setRange(newRange);
    onChange?.(newRange);
  };

  return (
    <div className="date-range-picker">
      <input
        type="date"
        value={range.start}
        onChange={handleStartChange}
        max={range.end}
      />
      <input
        type="date"
        value={range.end}
        onChange={handleEndChange}
        min={range.start}
      />
    </div>
  );
};

// Usage
const [dateRange, setDateRange] = useState({ start: '', end: '' });
<DateRangePicker {...dateRange} onChange={setDateRange} />
<template>
  <div class="date-range-picker">
    <input
      type="date"
      v-model="range.start"
      :max="range.end"
      @change="handleChange"
    />
    <input
      type="date"
      v-model="range.end"
      :min="range.start"
      @change="handleChange"
    />
  </div>
</template>

<script>
export default {
  props: {
    modelValue: { type: Object, default: () => ({ start: '', end: '' }) }
  },
  data() {
    return {
      range: { ...this.modelValue }
    }
  },
  methods: {
    handleChange() {
      this.$emit('update:modelValue', this.range);
    }
  }
}
</script>
<script>

Best Practices

When to Use Date Range Pickers

  • Data Filtering: Perfect for filtering analytics, reports, or logs by date ranges. Users can quickly select time periods to view relevant data.
  • Booking and Reservations: Essential for hotel bookings, car rentals, vacation planning, or any service requiring start and end dates.
  • Event Scheduling: Use for scheduling events, meetings, or activities that span multiple days.
  • Report Generation: Ideal for financial reports, performance reviews, or any report that requires a date range parameter.
  • Historical Analysis: When users need to compare data across different time periods or analyze trends.

When NOT to Use Date Range Pickers

  • Single Date Selection: If you only need one date, use a regular date picker instead. Don't force users to select the same date twice.
  • Fixed Periods: When only predefined periods are allowed (e.g., quarters, semesters), use a dropdown or radio buttons instead.
  • Real-time Data: For live dashboards showing current data, avoid forcing date range selection. Provide it as an optional filter.
  • Simple Time Periods: If you only need "this week" or "this month", consider using preset buttons instead of a full date range picker.

Design Guidelines

Visual Range Indication

Always highlight the selected range visually in the calendar. Users should see at a glance which dates are included in their selection.

Start and End Labels

Clearly label which date is the start and which is the end. Use descriptive placeholders like "Check-in" and "Check-out" or "From" and "To".

Range Validation

Automatically ensure the end date is after the start date. If users select dates in the wrong order, swap them automatically rather than showing an error.

Quick Presets

Include commonly used presets like "Last 7 Days", "Last 30 Days", "This Month" to speed up selection for frequent use cases.

Two-Month View

For desktop displays, show two months side-by-side to make it easier to select ranges that span month boundaries.

Accessibility Guidelines

  • Keyboard Navigation: Support full keyboard navigation including:
    • Tab to move between start/end inputs and calendar controls
    • Arrow keys to navigate between days
    • Enter/Space to select dates
    • Escape to close the dropdown
  • Screen Reader Support: Provide clear ARIA labels for all interactive elements. Announce range selection and changes to screen reader users.
  • Focus Management: When opening the calendar, move focus to the first selectable date or the current selection.
  • Visual Indicators: Use distinct colors and styles to indicate selected range, hover states, and disabled dates. Ensure sufficient color contrast.
  • Touch Targets: Make date cells large enough for easy touch interaction (minimum 44x44px).

Common Patterns

Analytics Dashboard

Combine with quick presets (Last 7 Days, Last 30 Days, This Month) for rapid date range changes. Position prominently at the top of the dashboard.

Booking System

Use clear labels like "Check-in" and "Check-out". Show the number of nights/days selected. Disable unavailable dates based on inventory.

Report Generation

Include presets for common reporting periods (This Quarter, Last Quarter, This Year). Display the selected range clearly before report generation.

Data Filtering

Allow clearing the filter to show all data. Provide a way to save or bookmark frequently used date ranges.

Event Planning

Show a duration indicator (e.g., "5 days selected"). Consider maximum range limits for practical planning.

Technical Implementation

Date Validation

Implement min/max date restrictions when needed. Validate that end date is after start date. Handle timezone considerations for multi-timezone applications.

State Management

Store both start and end dates separately. Clear both dates when user clicks Clear. Maintain selection state when calendar is closed and reopened.

Mobile Considerations

On mobile, consider showing one month at a time with swipe gestures for navigation. Ensure the dropdown doesn't overflow the viewport.
