Best Practices

When to Use

  • Multiple Options (6+): Select components work best when you have 6 or more options to choose from. They conserve screen space by hiding options until needed.
  • Known Options: Use selects when users are familiar with the options (e.g., country, state, month) and don't need to see all choices at once.
  • Single Selection: When users need to choose exactly one option from a predefined list.
  • Forms and Settings: Ideal for form fields, configuration panels, and settings pages where space efficiency is important.
  • Categorized Options: When options can be logically grouped using optgroup (e.g., programming languages by category, countries by region).
  • Mobile-Friendly: Native select elements provide excellent mobile experiences with optimized OS-level pickers.

When NOT to Use

  • Searchable Lists: If users need to search through many options (100+ countries, all cities), use a combobox or autocomplete component instead. Searching through a long select dropdown is frustrating.
  • 3-5 Options: For fewer options, use radio buttons instead. Radio buttons make all options visible and are faster to scan and select.
  • 2 Options: Use a switch/toggle or checkbox for binary choices (yes/no, on/off).
  • Complex Items: When options need rich formatting (images, multiple lines of text, custom layouts), use a custom dropdown component.
  • Multiple Selection: Native select elements with multiple have poor UX. Use checkboxes, multi-select component, or tag input instead.
  • Primary Actions: Don't use selects for primary actions or navigation. Use buttons or links.

Accessibility

  • Labels: Always provide a visible <label> element associated with the select using the for attribute or by wrapping the select.
  • ARIA Labels: If a visible label isn't possible, use aria-label or aria-labelledby to provide an accessible name.
  • Required Fields: Use the required attribute and indicate required fields visually (e.g., asterisk).
  • Error States: Use aria-describedby to associate error messages with the select. Apply select-error class for visual indication.
  • Keyboard Navigation: Native select elements are fully keyboard accessible:
    • Tab - Focus the select
    • Space or Enter - Open the dropdown
    • Arrow Keys - Navigate options
    • Escape - Close without selecting
    • Type to search - Jump to options starting with typed letters
  • Focus Indicators: Ensure focus states are clearly visible. Don't remove focus outlines.
  • Helper Text: Use helper text to provide additional context or instructions. Link it with aria-describedby.
  • Placeholder Option: The first option can serve as a placeholder (e.g., "Select an option..."). Give it an empty value so it can be validated as "not selected".

Common Patterns

Country Selector

Group countries by region using optgroup for easier scanning. Use ISO country codes as values. Consider providing a search functionality for long lists (use combobox instead).

Timezone Selector

Display timezones with GMT offset and city/region name for clarity. Group by major regions or sort by offset. Pre-select the user's detected timezone when possible.

Date Format

Show examples of how dates will appear (e.g., "MM/DD/YYYY (US)"). Include common regional formats. Respect user locale preferences.

Subscription Plans

Include pricing information directly in the option text. Order from free/lowest to highest price. Consider using a custom component if plans need detailed descriptions.

Settings and Preferences

Use selects for settings with predefined options (language, theme, notifications frequency). Provide helper text explaining what each setting affects.

Filtering and Sorting

Use selects for sort order (A-Z, newest first, price low-high) and filter criteria. Update content immediately or provide an "Apply" button for multiple filters.

Validation

  • Use the required attribute for mandatory selections
  • Provide immediate validation feedback when possible
  • Show clear error messages below the select using helper text
  • Apply select-error class for visual error indication
  • Don't remove the placeholder option until a valid selection is made

Styling Considerations

  • Consistent Sizing: Use size modifiers (select-sm, select-lg) consistently across your forms
  • Width: Set appropriate widths based on content. Don't make selects unnecessarily wide
  • Mobile: Native selects provide better mobile UX. Don't override with custom dropdowns on mobile unless necessary
  • Icons: When using icons, ensure adequate padding so text doesn't overlap
